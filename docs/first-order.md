주식 자동 거래 Agent를 만들려고 한다.
차트의 패턴을 보고 매매를 결정한다.
GCP의 Firebase functions에서 돌아간다. DB는 Firestore를 사용한다.

아래 내용들을 고려해야 한다.
- 주기적으로 다양한 값들을 체크하며 결정하는 구조를 생성해야 한다. cron과 같이 스케쥴링 되는 구조를 설정하라.
- 실제로 매매를 일으키기 전에 가상의 돈으로 시뮬레이션이 가능하도록 하라.
- 잘 알려진 패턴을 조사해 인식하는 방법들을 개별적으로 정리해 함수로 만든다.
- firebase의 무료 요금제 범위 안에서 작동하도록 하라. Firebase가 적절하지 않으면 GCP Cloud Run을 제안하라.
- 종목을 스캔하는 스케쥴링의 전략을 설계하라.
- 개별 전략들의 사용에 대한 우선순위를 정하는 전략을 설계하라.
- 사용자가 개별 전략들의 우선순위에 직접적으로 개입할 수 있는 방법을 제공하라.
- 전체 Master MD 파일을 작성하고, 그 안에서 기능별, 전략별로 개별 MD파일을 작성한다.
- MD 파일 하나는 500줄을 넘기지 않는다.

전체적으로 Agent 기반으로 작동하게 하되, 아래 사항들을 고려해 설계한다.
- 전략상 LLM을 이용해야 하는 경우 이를 위한 외부 LLM API 주소를 설정 파일에 설정할 수 있도록 하라.
- 무엇을 기반으로 Agent가 결정할지를 개별 Agent마다 구체적으로 정의하라.
- Agent들은 LLM을 이용하지 않아도 결정할 수 있는 알고리즘을 포함해야 한다.
- 가능하다면 LangGraph를 이용하라. 적절하지 않다면 사용하지 않아도 된다.
- LangGraph가 가능하다면 이 프로그램의 노드별 작동 상황을 그래프 이미지 상에서 눈으로 확인할 수 있도록 하는 환경을 구성하라. 즉, admin화면에 접속하면 실시간으로 작동양상이 그래프형 GUI 상에서 표현되어 작동되고 있는 node의 점멸, 작동 로그가 확인될 수 있도록 한다.

마지막으로, 각종 설정값들을 외부에서 설정해서 실행되도록 설정파일을 만들어 참조하도록 한다.

실제로 구현하지는 말고 docs/plans에 설계 md 파일을 만들어라.
전체 프로젝트에 대한 간략한 설명과 중요한 결정 및 실행 방법을 /README.md에 기재하라.
깊이 생각하고 실행하라. 


추가 고려 사항
- 가능한 공개 API를 이용한다.
- 공개 API를 쓰지 못하는 경우와 매매의 경우 삼성 증권의 API를 이용해 정보를 수집하게 한다.
- 시뮬레이션 모드에서는 실제 매매를 하지 않는다. 설정에서 시뮬레이션 모드를 선택할 수 있어야 한다.
